# 商资通 H5 稳定化与部署设计

- 日期：2026-07-20
- 状态：已完成方案确认，待文档复核
- 视觉方向：B · 资金指挥舱
- 部署目标：`47.253.226.91/szt/`

## 1. 背景与目标

商资通现有后端采用 FastAPI，前端为微信小程序。由于小程序和正式域名仍在注册，本阶段先交付手机、桌面浏览器均可使用的 H5。

47 服务器的根路径已承载其他系统，商资通必须继续使用 `/szt/`，不得影响根站点。当前 FastAPI 由人工启动，服务器上的单文件 H5 原型未进入 Git，且存在前后端字段错配。本阶段目标为：

1. 修复资金缺口重复扣减、同日结算覆盖等 P0 问题。
2. 建立唯一的 30 天每日资金账本，统一日历、调度、提醒和止损口径。
3. 在 `frontend/h5/` 新建 React + Vite + TypeScript H5。
4. 通过 HTTPS 和访问密码提供临时受保护访问。
5. 将后端纳入 systemd，并建立备份、健康检查及回滚流程。
6. 保留 `frontend/miniprogram/`，为以后微信小程序接入留出路径。

本阶段不重写后端、不强制迁移数据库、不开放邮件/SFTP、不实现正式微信授权登录，也不替换服务器根站点。

## 2. 方案选择

采用“定向稳定化 + 独立 H5 + 保守部署”：

- 保留 FastAPI 和 SQLite，只修复核心正确性、安全边界与运行方式。
- H5 与小程序并存，使用明确的 TypeScript 接口模型。
- Nginx 在 `/szt/` 提供 H5，在 `/api/v1/` 代理后端。
- 本机验证后推送 GitHub，47 服务器只同步指定的已验证提交。

不采用全面重构和数据库迁移，因为会扩大数据风险并延迟上线；不继续修补服务器单文件 H5，因为它不可测试且字段映射已经失配；不先做前端后修后端，因为资金工具不能展示已知错误的缺口结果。

## 3. 总体架构

```text
手机/桌面浏览器
       |
       v
Nginx（HTTPS、访问密码）
  | /szt/                 | /api/v1/
  v                       v
React H5            FastAPI（127.0.0.1:8800）
                            |
             +--------------+--------------+
             |              |              |
       统一资金账本      现有算法服务       SQLite
             |
       日历 / 调度 / 止损 / 提醒 / 报告
```

后端只监听 `127.0.0.1:8800`。所有公网请求必须经过 Nginx，避免绕过访问保护与可信身份头。

## 4. 统一资金流

### 4.1 当前可用资金

H5 增加非必填的“当前可用资金”，表示商户可用于经营调度的银行卡、现金等资金总额。新增一对一商户资料模型：

| 字段 | 规则 |
|---|---|
| `user_id` | 唯一，对应当前商户 |
| `available_cash` | Decimal，可空；空值与明确填写 0 元必须可区分 |
| `available_cash_updated_at` | 最近一次余额更新时间，可空 |
| `created_at` / `updated_at` | 审计时间 |

未填写时，计算服务以 0 元临时试算，但不把空值保存成 0。接口返回 `is_estimate=true` 和未设置状态；H5 明确显示“试算”和设置入口。修改后立即重算，并展示最后更新时间。

### 4.2 每日资金账本

```text
当日期末余额 = 当日期初余额
             + 当日预计结算入账
             - 当日信用卡还款
             - 当日计划进货
             - 其他已确认支出

当日资金缺口 = max(0, -当日期末余额)
次日期初余额 = 当日期末余额
```

规则：

1. 同一天多笔结算按日期求和，禁止使用日期键覆盖单笔数据。
2. 先完成当日全部收支，再根据期末余额判断缺口。
3. 缺口只取负余额绝对值，不再次扣除已进入账本的还款。
4. 日历、调度、止损、提醒和报告必须调用统一服务，不得复制计算逻辑。
5. 货币使用 Decimal，并在 API 边界统一格式化。

每日输出包括日期、期初余额、各类收支、期末余额、缺口、事件、数据完整性、是否试算及资金更新时间。

### 4.3 API

- `GET /api/v1/profile/cash`：读取当前可用资金。
- `PUT /api/v1/profile/cash`：设置、修改或清空当前可用资金。
- `GET /api/v1/cashflow?days=30`：返回统一账本和汇总。
- `POST /api/v1/auth/session`：使用 Nginx 可信身份建立 H5 会话。

现有日历、调度、止损、提醒和报告接口保持兼容，内部改为调用统一服务。经营报告只使用真实账本事件，不再把代理指标表述为真实逾期或资金缺口。

## 5. H5 信息架构

移动端使用 5 个底部入口，桌面端切换为左侧导航：

- 首页：资金概览、30 天时间轴、最低余额、风险和待办。
- 资金：当前可用资金、信用卡、流水上传。
- 规划：进货推荐、全局调度、止损方案。
- 提醒：还款提醒、预计缺口、处理状态。
- 我的：经营健康报告、数据状态、退出访问。

邮件和 SFTP 在“资金”页以禁用卡片展示“正在开发中”，不能进入配置表单。

首次使用采用可跳过的轻量引导：

```text
访问验证
  -> 当前可用资金（可跳过）
  -> 添加信用卡
  -> 上传结算流水
  -> 生成资金指挥舱
```

空数据页面解释下一步操作，不展示容易误解的全零图表。

## 6. 视觉系统

采用“B · 资金指挥舱”：

- 深墨绿背景和高对比浅色文本，强调稳健、专业。
- 青绿表示正常资金，琥珀表示待关注，红色只用于明确缺口和高风险。
- 首页优先展示决策信息，不堆叠装饰图表。
- 金额、日期、状态保持稳定对齐和层级，移动端支持单手操作。
- 加载、空、错、试算均有明确状态。

## 7. 临时认证与安全

无正式域名期间采用两层保护：

1. Nginx Basic Auth 保护 `/szt/` 和 `/api/v1/`。
2. H5 通过会话接口把 Nginx 的 `$remote_user` 映射为商户身份，后端下发应用会话 Cookie。

安全要求：

- Nginx 覆盖 `X-Authenticated-User`，不透传客户端伪造值。
- 后端只接受本机 Nginx 的可信身份头。
- Cookie 使用 `HttpOnly`、`Secure`、`SameSite=Lax` 和合理有效期。
- 修改类请求校验同源 `Origin` 或 CSRF 令牌。
- CORS 限制同源，HTTP 的商资通路径跳转 HTTPS。
- 密钥和数据库路径通过环境变量提供，不进入 Git。
- 日志不记录密码、令牌、完整手机号或上传原文。

邮件/SFTP的 H5 配置入口和后端路由默认关闭，不新增明文凭据。后续启用前必须完成凭据加密、密钥管理与日志脱敏。

上传限制：

- 单文件不超过 10MB。
- 同时校验扩展名和实际内容类型。
- 服务端生成文件名，禁止使用用户文件名拼接路径。
- 预览会话绑定所有者、有效期和一次性确认。
- 逐行返回可理解的解析错误。

## 8. 异常处理

- 网络或接口失败：保留页面和输入，提供重新加载。
- 未授权：清理应用会话并重新访问验证。
- 空数据：提供添加卡片或上传流水的下一步。
- 未设置当前可用资金：显示“试算”，不表述为确定缺口。
- 余额长期未更新：展示更新时间和更新提示。
- 上传先预览后入库，错误行单独说明，不整批静默失败。
- 后端统一返回错误码、用户消息和请求追踪标识。
- H5 不展示堆栈、SQL 或内部路径。

## 9. 部署

### 9.1 Nginx 与进程

- 保留根站点；只调整 `/szt/` 与 `/api/v1/`。
- `/szt/` 指向版本化 H5 当前发布目录。
- `/api/v1/` 代理至 `127.0.0.1:8800`。
- 配置访问保护、可信用户头、上传限制和代理超时。
- Nginx 修改先做语法检查，再平滑重载。
- 建立 `szt-backend.service`，使用项目虚拟环境和受保护环境文件，支持开机启动和失败自动重启。

### 9.2 发布流程

```text
本机测试和构建
  -> 推送 GitHub 指定提交
  -> 服务器检查工作区与磁盘
  -> 备份 SQLite 和当前 H5
  -> 快进同步代码
  -> 安装锁定依赖并构建 H5
  -> 切换 H5 版本
  -> 重启后端
  -> 页面/API/核心流程健康检查
  -> 失败则恢复上一版本
```

数据库变化采用向后兼容的小型迁移。发布不得覆盖服务器其他业务。
